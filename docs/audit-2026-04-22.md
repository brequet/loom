# Loom Code Audit Report
Generated: 2026-04-22

## BUGS

### B1 - `orchestration.rs:12` - Panic on `.unwrap()` on `opencode_port`
`session.opencode_port` is `Option<i64>`. If it is ever None (DB corruption, future code path), this panics the handler.
Fix: use `.ok_or(AppError::...)? ` instead.

### B2 - `services/session.rs:186` - `terminate_session` bypasses state machine
Directly writes `"terminated"` without going through `update_state`, which validates transitions.
A session in any state (including `provisioning`) can be terminated, inconsistent with the state machine.
Fix: apply state guard or document the bypass explicitly.

### B3 - `routes/sessions.rs:99` - Stopping a `provisioning` session is a race condition
If `create_session` becomes async/non-blocking in the future, stopping a provisioning session while orchestration is running could cause undefined state.
Currently safe because orchestration is synchronous, but a latent design inconsistency.

### B4 - `NewSessionDialog.svelte:88` - `?autoOpen=1` is fragile in hash router
```ts
push(`/sessions/${session.id}?autoOpen=1`);
```
In svelte-spa-router, `?` appends to the hash fragment, not as a proper query param. Works coincidentally because detection uses `window.location.hash` directly, not router params. Should use a more reliable mechanism (e.g. store or `#/sessions/${id}#autoOpen=1`).

## DEAD CODE

### B5 - `services/gitlab.rs:62` - `get_merge_request()` never called
Implemented but unused. `#[expect(dead_code)]` silences the warning. Wire it up or remove it.

### B6 - `error.rs:46,57` - Unused error variants
`OpenCodeSessionCreate` and `JiraApi` variants are defined and matched but never constructed.
Remove or use them.

### D1 - `shared/SessionListResponse.ts` - Never imported in frontend
The API client (`sessions.ts`) uses an inline type `{ sessions: Session[] }` instead of the generated type.

### D2 - `shared/CreateSessionRequest.ts` - Never imported in frontend
`createSession()` uses a manually defined inline type. The generated type goes unused.

### D6 - `jira.ts:4`, `gitlab.ts:4` - Unnecessary type re-exports
Types are already available from `$shared/*` directly. No component imports them from `$lib/api/`.

## MINOR / INCONSISTENCIES

### M1 - `models.rs:36,68` - Silent fallback in `from_str_val`
Unknown DB strings silently map to `SessionState::Stopped` or `SourceType::Scratch`.
Should log a warning or return `Result` to surface DB corruption.

### M3 - `opencode.rs:65` - Hardcoded `localhost` for opencode calls
```rust
let url = format!("http://localhost:{port}/session");
```
Breaks if opencode runs on a different host (container, remote). Add `LOOM_OPENCODE_HOST` config.

### M4 - `main.rs:89` - CORS fully open
```rust
CorsLayer::new().allow_origin(Any).allow_methods(Any).allow_headers(Any)
```
Acceptable for a local tool, but should be restricted if ever exposed remotely.

### M5 - `SessionDetail.svelte:14` - `autoOpened` should be `$state`
Plain `let autoOpened = false` mutated inside `$effect`. Works by accident (it's only a guard, not displayed). Should be `$state(false)` for Svelte 5 correctness.

### M6 - `Dashboard.svelte:39` - `terminated` in sort order but never returned
Backend filters `WHERE state != 'terminated'` so terminated sessions never appear. The `terminated: 3` entry in `stateOrder` is dead.

### M7 - `NewSessionDialog.svelte:116` - GitLab sessions get fabricated titles
Jira sessions fetch the real issue summary for the title. GitLab sessions use `MR !{iid}`. Inconsistent UX.

### M8 - `opencode_config.rs:44` - JSONC injection fragile for nested configs
`inject_loom_entry` finds the first `}` after `{` -- breaks if `permission` or `external_directory` contain nested objects. Acceptable for the narrow current use case but risky for complex user configs.

### M9 - `prompt.rs:143` - Double-negative, use `is_some_and`
```rust
// Current (hard to read):
!jira_issue.as_ref().is_none_or(|i| i.components.is_empty())
// Better:
jira_issue.as_ref().is_some_and(|i| !i.components.is_empty())
```

### M10 - `db.rs:13` - SQLite without WAL mode
Multiple connections without WAL serialize writes. Set `PRAGMA journal_mode=WAL` after connect.
